## 객체 지향 설계와 스프링

- 스프링 프레임워크
- 스프링 부트
	* 스프링을 편리하게 사용할 수 있도록 지원 
	* 스프링 프레임워크 기반으로 작동
- 스프링
	* 스프링 DI 컨테이너 기술 
	* 스프링 부트, 스프링 프레임워크 등을 모두 포함한 스프링 생태계
	* 스프링은 자바 언어 기반의 프레임워크
	* 자바 언어의 가장 큰 특징 - **객체 지향 언어** 
	* 스프링은 객체 지향 언어가 가진 강력한 특징을 살려내는 프레임워크
	* 스프링은 **좋은 객체 지향** 애플리케이션을 개발할 수 있게 도와주는 프레임워크
- 좋은 객체 지향 프로그래밍
	* 다형성
		*  다형성의 본질
			* 인터페이시를 구현한 객체 인스턴스를 실행 시점에 유연하게 변경할 수 있음
			* 즉 클라이언트를 변경하지 않고, 서버의 구현 기능을 유연하게 변경할 수 있음 
		*  다형성의 실세계 비유
			1. **역할**과 **구현**으로 세상을 구분
			2. 운전자와 자동차 역할 
			3. K3, 아반떼, 테슬라 모델3 ...등으로 구현
			4. 구현된 자동차는 역할만 수행하면 운전자는 그대로 운전할 수 있음
			5. 결국 자동차 역할의 구현은 운전자(Client)에게 **영향을 주지 않고 새로운 기능**을 제공

	* 5가지 원칙 SOLID
		1. SRP 단일 책임 원칙 	
		2. **OCP 개발-폐쇄 원칙**
			* 확장에는 열려 있고 변경에는 닫혀 있음
			(자동차 역할은 고정되어 있지만 새로운 모델은 얼마든지 구현할 수 있음)
		3.  LSP 리스코프 치환 원칙
			* 프로그램의 정확성을 지킴 
			(자동차의 엑셀은 앞으로 가는 기능으로 제한)
		4. ISP 인터페이스 분리 원칙
			* 특정 클라이언트를 위한 인터페이스 여러 개가 한 개보다 낫다 
		5. **DIP 의존관계 역전 원칙**
			* 프로그래머는 추상화에 의존해야지, 구체화에 의존하면 안됨
			(운전자는 아반떼가 어떻게 구현 됐는지가 아니라 자동차 역할만 알면 된다)

- 객체 지향 설계와 스프링
	* 스프링은 DI 컨테이너로 OCP, DIP 원칙을 지키게 해줌 

- 정리 
	1. 모든 설계에 역할과 구현을 분리 (자동차 사례를 참고로)
	2. 애플리케이션도 자동차처럼 역할만 만들어두고, 모델은 유연하게 구현할 수 있는게 좋은 객체 지향 설계
	3. 실무 고민
		* 인터페이스 도입은 추상화라는 비용 발생
		* 기능 확장할 가능성이 없다면, 구체 클래스를 직접 사용하고, 향후 꼭 필요할 때 리팩터링해서 인터페이스를 도입
